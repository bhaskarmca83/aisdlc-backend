# aisdlc-backend

Spring Boot 3 / Java 21 sample target application. This is the codebase that the AI SDLC pipeline writes code into — ImplementationAgent opens feature branches here, TestAgent runs the test suite here, and ReviewAgent reviews PRs raised against this repo.

It also serves as the reference brownfield application used when demonstrating the platform against an existing Java stack.

## Role in the Pipeline

When a pipeline run targets this repo:

1. **ImplementationAgent** checks out a feature branch, generates code into the appropriate package (`auth/`, `common/`, `config/`), and commits.
2. **TestAgent** runs `mvn test` via a shell tool call; result (pass/fail + log excerpt) goes into `SDLCState.test_result`.
3. **ReviewAgent** reads the diff with `review_rules` from the registered repo's tech profile (typically `spring-boot-java`) and produces a verdict.
4. **DeployAgent** (if configured) builds the JAR and pushes to ECS or triggers a GitHub Actions workflow.

The profile assigned to this repo in the project registration drives all of the above. Default profile: `spring-boot-java`.

## Local Setup

**Prerequisites:** Java 21+, Maven 3.9+, PostgreSQL 16 running locally

```bash
# Start PostgreSQL (or reuse the docker-compose from aisdlc-orchestrator)
# docker compose -f ../aisdlc-orchestrator/docker-compose.yml up -d postgres

# Create database
psql -U postgres -c "CREATE DATABASE aisdlc_backend;"

# Configure datasource
export SPRING_DATASOURCE_URL=jdbc:postgresql://localhost:5432/aisdlc_backend
export SPRING_DATASOURCE_USERNAME=postgres
export SPRING_DATASOURCE_PASSWORD=postgres

# Run
mvn spring-boot:run
# Starts at http://localhost:8080
```

Application properties are in `src/main/resources/application.yml`.

## Run Tests

```bash
mvn test
```

Tests use Testcontainers — Docker must be running. Each test suite spins up a real PostgreSQL container and tears it down on completion.

**Do not mock the database.** Prior incidents showed that mock/real divergence masked broken schema migrations. Testcontainers is the mandated approach.

## API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/auth/register` | None | Register a new user |
| `POST` | `/api/auth/login` | None | Authenticate; returns JWT |
| `GET` | `/api/health` | None | Liveness check |
| `GET` | `/api/users/me` | Bearer JWT | Get current user profile |

Additional endpoints are generated by ImplementationAgent per story and follow the same package structure.

## Key Architectural Decisions

**Why Spring Boot?** It's the most common Java enterprise framework and the most likely brownfield target for teams using this platform. Using it as the sample makes ImplementationAgent's code generation realistic for the majority of Java shops.

**Why Testcontainers instead of H2?** Schema migrations, column types, and constraint behavior differ between H2 and PostgreSQL in ways that break in production. The overhead of spinning up a real container per test suite is acceptable; the risk of masked failures is not.

**Why JWT over sessions?** The orchestrator's ImplementationAgent writes stateless REST APIs by default. Stateless auth aligns with the ECS/EKS deployment target where there is no sticky session affinity.

## Extending This Application

When the pipeline adds a new feature:
- New endpoints go in a new package under `com.aisdlc.<domain>/`
- DTOs go in `common/dto/` or domain-local `dto/`
- New entities go in `<domain>/model/` with a Flyway migration in `resources/db/migration/`
- Tests mirror the source tree: `src/test/java/com/aisdlc/<domain>/`
